`npx` is a small command with a useful promise: run an npm package executable without making you install that tool into your project first.

That makes it especially handy for project generators, formatters, migration tools, and quick checks. You can run a command once, get the result you need, and keep your `package.json` and local `node_modules` free of a dependency you did not intend to keep.

## The Short Version

When you run:

```bash
npx create-vite@latest my-app
```

the rough flow is:

```text
resolve package -> download if needed -> run package command -> leave project dependencies alone
```

From the project's point of view, the package can look like it never existed. It is not added to `package.json`, and it is not installed into the project's normal dependencies just because you used `npx`.

There is one important caveat: npm may keep downloaded package data in its cache. So `npx` is temporary with respect to your project, but not necessarily erased from your machine immediately after the command finishes.

## Why npx Exists

Before `npx`, if you wanted to run a package-provided command, you often had to install it first:

```bash
npm install --save-dev some-tool
npm run some-tool
```

or install it globally:

```bash
npm install --global some-tool
some-tool
```

Both approaches are useful in the right context. But sometimes you only need a tool once:

```bash
npx create-vite@latest my-app
```

That is where `npx` shines. It gives you a quick path from "I need this command" to "run it now."

## npx and npm exec

In modern npm, `npx` is closely related to `npm exec`. You can think of it as the convenient command people commonly reach for when they want to execute a package binary.

These are conceptually similar:

```bash
npx create-vite@latest my-app
npm exec -- create-vite@latest my-app
```

The exact spelling matters less than the mental model: npm resolves a package, finds the executable it exposes, and runs it with the arguments you provide.

## What the `@tag` Part Means

You will often see commands like this:

```bash
npx package-name@tag
```

The `@tag` part tells npm which published version or release channel to use.

Common examples:

| Command | Meaning |
| --- | --- |
| `npx create-vite@latest` | Use the package version tagged as `latest` |
| `npx some-tool@next` | Use the package version tagged as `next`, if it exists |
| `npx some-tool@beta` | Use the beta release channel, if the package publishes one |
| `npx some-tool@1.2.3` | Use exactly version `1.2.3` |

If you omit the tag:

```bash
npx package-name
```

npm commonly resolves the package's `latest` version, unless a matching local executable is already available and npm chooses that local command.

For repeatable instructions, including the tag is often clearer:

```bash
npx create-vite@latest my-app
```

For maximum repeatability, pin an exact version:

```bash
npx some-tool@1.2.3
```

## Where the Arguments Go

In this command:

```bash
npx create-vite@latest my-app --template react
```

the package command is:

```text
create-vite@latest
```

and the arguments passed to that command are:

```text
my-app --template react
```

So yes: after the package name and tag, the remaining values are normally arguments for the package executable.

That distinction matters because `npx` also has its own options. Options for `npx` should go before the package command:

```bash
npx --yes create-vite@latest my-app --template react
```

Here, `--yes` belongs to `npx`, while `my-app --template react` belongs to `create-vite`.

When a command gets ambiguous, use `--` as a separator:

```bash
npx --package cowsay -- cowsay hello
```

The separator helps mark the boundary between npm's execution options and the command you want to run.

## Is npx Installing the Package?

The honest answer is: it depends what you mean by "installing."

`npx` may download package contents so the command can run. But it does not automatically add that package as a dependency of your current project.

That means this:

```bash
npx prettier@latest . --check
```

does not mean your project now depends on `prettier`.

But npm may still cache the package under npm's cache directory. That cache helps make future runs faster. If you want to clear npm's cache manually, npm provides:

```bash
npm cache clean --force
```

Use that carefully; the cache is usually helpful.

## A Useful Rule of Thumb

Use `npx` when:

- you need to run a package command once;
- you are following a scaffolding command from a tool's docs;
- you want to try a CLI before deciding whether to add it to your project;
- you want to run a specific published version without changing your dependency file.

Install the package locally when:

- the project needs the tool repeatedly;
- the command belongs in `npm scripts`;
- your team needs the same version in development or CI;
- reproducibility matters more than convenience.

For example, this is fine for a one-off:

```bash
npx eslint@latest .
```

But a real project usually benefits from a local dev dependency:

```bash
npm install --save-dev eslint
npm exec -- eslint .
```

That way the version is recorded and shared with everyone working on the project.

## The Security Footnote That Matters

`npx` makes remote package execution easy. That is powerful, but it also means you should be thoughtful about what you run.

Before running an unfamiliar package:

- check that the package name is spelled correctly;
- prefer official documentation for install commands;
- consider pinning an exact version for repeatable setup;
- avoid running commands from packages you do not trust.

`npx` is convenient, not magic protection. It runs code from npm packages, and package code can do real work on your machine.

## The Mental Model

The cleanest way to remember `npx` is:

```text
npx = run this npm package executable now
```

The `@tag` selects the version or release channel. The arguments after the package name usually belong to the package command. The package does not get added to your project just because you ran it, though npm may keep a cached copy behind the scenes.

That is the whole trick: quick execution, clean project state, and just enough version control when you add `@latest`, `@beta`, or an exact version.
