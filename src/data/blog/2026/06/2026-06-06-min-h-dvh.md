`min-h-dvh` is a Tailwind CSS utility class. It is short, but it contains three separate ideas:

```css
min-height: 100dvh;
```

To understand it clearly, it helps to split it into two parts:

- `min-height`
- `dvh`

## What min-height means

The CSS `min-height` property sets the minimum height an element is allowed to have.

```css
.panel {
  min-height: 300px;
}
```

That means `.panel` will be at least `300px` tall.

If the content inside the element is shorter than `300px`, the element still keeps that minimum height. If the content is taller than `300px`, the element can grow.

This is the key difference between `height` and `min-height`:

| Property | Meaning |
| --- | --- |
| `height: 300px` | Try to make the element exactly `300px` tall |
| `min-height: 300px` | Make the element at least `300px` tall, but let it grow |

So `min-height` is useful when you want a layout to fill enough space without trapping content inside a fixed box.

## What the viewport is

The viewport is the visible area of the web page inside the browser window.

On a laptop, it is the area inside the browser tab where the page is rendered. On a phone, it is the visible part of the web page on the screen. If you resize the browser window, the viewport changes.

CSS viewport units are based on that visible browser area:

```css
height: 100vh;
```

`vh` means viewport height. `100vh` means 100% of the viewport height.

## Why dvh exists

At first glance, "viewport" already sounds dynamic. If the visible browser area changes, should the viewport not change too?

In ordinary language, yes. But in CSS history, especially on mobile browsers, `vh` has not always behaved like the current visible area.

Mobile browsers have interface chrome: address bars, tab bars, bottom toolbars, and other controls. These controls can appear and disappear while the user scrolls. When that happens, the truly visible web page area changes.

Traditional `100vh` often behaved more like a stable layout reference than the exact current visible area. On some mobile browsers, an element with `height: 100vh` could become too tall when the address bar was visible, causing awkward overflow or hidden content.

Modern CSS adds more precise viewport units:

| Unit | Meaning |
| --- | --- |
| `svh` | Small viewport height, often when browser UI is expanded |
| `lvh` | Large viewport height, often when browser UI is collapsed |
| `dvh` | Dynamic viewport height, the current changing viewport height |

So a practical mental model is:

```text
vh  = older viewport height; may be awkward on mobile
dvh = current dynamic visible viewport height
```

This is not exactly "a bug fix" as much as a clearer vocabulary. CSS now lets authors choose which viewport behavior they actually want.

## What min-h-dvh means in Tailwind

Tailwind's `min-h-dvh` utility maps to:

```css
min-height: 100dvh;
```

In plain English:

> Make this element at least as tall as the current visible viewport.

That is useful for full-screen application shells, readers, dashboards, centered screens, and layouts that should fill the visible page without using a brittle fixed height.

For example:

```html
<main class="min-h-dvh">
  ...
</main>
```

The `main` element will be at least as tall as the current dynamic viewport. If its content needs more room, it can still grow taller because this is `min-height`, not `height`.

## Why not just use h-dvh

Tailwind also has height utilities, such as `h-dvh`, which maps to:

```css
height: 100dvh;
```

That says the element's height is `100dvh`.

Sometimes that is what you want, especially for a fixed-height app region where inner content scrolls. But for many page-level containers, `min-h-dvh` is more forgiving because it allows the page to grow when content is larger than the viewport.

Compare:

| Tailwind class | Rough CSS | Behavior |
| --- | --- | --- |
| `h-dvh` | `height: 100dvh` | The element is exactly the dynamic viewport height |
| `min-h-dvh` | `min-height: 100dvh` | The element is at least the dynamic viewport height and can grow |

If you are building a normal page container, `min-h-dvh` is often the calmer choice.

## When to use min-h-dvh

Use `min-h-dvh` when a section should fill the visible screen but still adapt to content.

Good examples include:

- A full-page reader layout
- A sign-in screen
- A page shell with a header and main content
- A centered empty state
- A tool or app surface that should occupy the visible browser area

For example:

```html
<main class="flex min-h-dvh items-center justify-center">
  <section>
    Sign in
  </section>
</main>
```

Here `min-h-dvh` gives the `main` element enough height to fill the visible screen, while `flex`, `items-center`, and `justify-center` handle alignment.

## The simple version

You can remember `min-h-dvh` like this:

```text
min-h-dvh = minimum height: the current visible screen height
```

Or slightly more technically:

```css
min-height: 100dvh;
```

It combines the flexibility of `min-height` with the mobile-aware behavior of dynamic viewport units. That makes it a strong default for modern full-screen-ish layouts, especially when the page needs to behave nicely on phones.
